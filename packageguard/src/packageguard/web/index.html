ARE YOUR ATTACK SURFACE.

A typical project pulls in hundreds of transitive dependencies you never chose and never read. Any one of them — or any future update to one — can compromise your entire application.

✚DEFENSE PROTOCOL · STAY SAFE
01

Check before you install

Inspect download count, publish history and maintainer first — especially for small single-purpose utilities.

▸ CHECK does this automatically
02

Always commit your lockfile

package-lock.json pins exact versions; without it a routine install can silently pull a compromised update.

▸ SCAN reads your lockfile
03

Watch transitive dependencies

The package you installed is rarely the risk — what it depends on is. Most devs never look past level one.

▸ GRAPH traces the whole tree
04

Rotate credentials after an incident

If a package you use is confirmed compromised, assume every secret on that machine (npm tokens, API/SSH keys) is exposed.

→ SCAN results include rotation steps
05

Distrust unfamiliar names

Typosquats (loadash vs lodash) still trick people under time pressure — one keystroke, wrong package.

→ The ML model catches look-alikes, DB or not
06

Minimize what you depend on

Every dependency is code you're trusting. Fewer of them = a smaller attack surface, full stop.

→ CHECK flags unusually heavy packages
// FINAL TRANSMISSION

No tool — ours included — catches everything. npm audit knows only disclosed CVEs. A malware database knows only names it's already seen. PackageGuard's graph model is built for what those miss: a threat hiding inside a dependency, before anyone has reported it. Layer your defenses. Stay skeptical of new dependencies.
